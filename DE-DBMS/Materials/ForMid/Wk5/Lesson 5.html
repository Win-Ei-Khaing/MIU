Lesson 5 - Normalization

Purpose: -   How to define relations to minimize the possibility of insertion, update, and deletion anomalies without recourse to software implementations of business logic.

Note: The normalization procedure for a relational database schema is really common sense, but the work that went in to describing the detailed structure of the common sense principles provided the basis for automation and validation software.

Terminology

A constraint on a relational database is a Boolean relation on a set of qualified attributes. The basis for a constraint is generally a business rule. For example, a constraint that an employee may only work in one department at a time is not a logical necessity but could be a rule in a particular business.
 
A database is in a consistent state if the data in the state does not violate any constraints.
For example, in a relation Staff(EmpName, Department, Location), there could be a constraint that each department has a unique location. If different tuples put a department in different locations, then the database is in an inconsistent state.
 
SQL checks or DBMS triggers can be used to enforce consistency, but good design of the database provides a more robust solution for enforcement of the simpler constraints.
 
Business rules can give rise to two types of constraints on the values of attributes in a relation. Let S1, S2, and S3 be sets of attributes of a relation R. Then:
        Functional dependency: S1→S2 if, in every instance of R, if two tuples have the same value of S1, then they also have the same value for S2. For example, S1=(ZipCode) functionally determines S2=(State). A functional dependency limits the possible tuples that can occur in R.
        Lossless join dependency: every instance of R is equal to the join of two or more projections of R. For example, if S1=(LoanNo), S2=(CustNo), S3=(CustEmail), and if a customer can have more than one loan and more than one email address, then every instance of R should be equal to the join of its projections R1(LoanNo,CustNo) and R2(CustNo,CustEmail). A lossless join dependency may require insertion of multiple rows where one would be expected.
 
The determinant of a functional dependency S1→S2 is the set of attributes S1.
 
In any relation, every attribute is functionally dependent on every superkey.
 
One cannot tell whether a relation R has a functional or lossless join dependency by inspecting instances of it. However, if an instance of R violates a proposed dependency, then we can conclude that the dependency does not hold for R.
 
An instance of a relation R is consistent if it satisfies all the constraints on R.
 
A relation R is a lossless join of two projections R1 and R2 of R if every consistent instance of R is the equi-join of its two projections into instances of R1 and R2.
 
If R1 and R2 are projections of a relation R, the equi-join of R1 and R2 will be lossless if and only if the attributes in common to R1 and R2 form a superkey for one of R1 or R2.
 
A relation R is a lossy join of two projections R1 and R2 of R if some consistent instance of R is a proper subset of the equi-join of its projections into instances of R1 and R2. The term lossy signifies a loss of information in the passage from R to the equi-join of R1 and R2.
 
A lossless join is trivial if the join matches attributes of candidate keys.
 

Normalization 

Normalization is a feature of a relation which automatically leads to data consistency with respect to most functional dependencies and all lossless join dependencies. The basic requirements for a normalized relation are that:
        All attributes describe the primary key as a whole
        Some instance of the relation cannot be expressed as the join of two or more projections of that instance

Purpose of Normalizing Data

The purpose of normalization is to identify a suitable set of relations that support the data requirements of an enterprise.
The characteristics of a suitable set of relations include the following:
• the minimal number of attributes necessary to support the data requirements of the enterprise;
• attributes with a close logical relationship (described as functional dependency) are found in the same relation;
• minimal redundancy with each attribute represented only once with the important exception of attributes that form all or part of foreign keys,which are essential for the joining
of related relations.
The benefit of using a database that has a suitable set of relations is that the database will be easier for the user to access and maintain the data, and take up minimal storage space on the
computer.

The Stages of Normalization

1NF: at most one fact in one location.
        Implied by the definition of a relation as a subset of a Cartesian product.
 
2NF: no partial dependencies
A FD S→X conforms to 2NF if either:
a.       S is not a proper subset of a candidate key, or
b.      X is an attribute in a candidate key
A FD S→X violates 2NF if both:
a.       S is a proper subset of some candidate key, and
b.      X is not an attribute in any candidate key
        For example, in a relation R(A,B,C), primary key (A,B), if functional dependency FD(B;C) then split R into two relations, R1(A,B) and R2(B,C).
 
3NF: no transitive dependencies
A FD S→X conforms to 3NF if (a) S contains X, or (b) S is a superkey, or (c) X is an attribute of a candidate key.
        For example, 2NF and not 3NF: relation R(A,B,C) with primary key A, and no alternate key, if FD(B;C) then split R into R1(A,B) and R2(B,C).

2.      A relation R of journal holdings has a schema R(Title, Keywords, Holdings)
 
A few tuples might look like:
 
Title Keywords Holdings
SIGSAM Algorithms Computing, algorithms Dec 1990 to Feb 1995
SIGSAM Algorithms Computing, algorithmics Jan 1998 to present
The Network Expert Computer Science, networks Sept 1999 to present
 
a)      What normal forms are violated?
ANS:  
1NF: Keywords is a repeating field, Holdings contains two separable data items
2NF: The only candidate key is (Title, Holdings), but there is a functional dependency (Title → Keywords). So we have a partial dependency of Keywords on the primary key, (Title, Holdings).

b)      What problems does this cause?
ANS:
1NF: There is no control over keywords, so the same intention may be represented by numerous variations. For example, algorithms or algorithmics, computing or computer science.
2NF: If we want to add a keyword for a journal title, we may have to add it on more than one row.

c)      Propose a normalized version of the relation.
ANS:
1NF: Repeat tuples so that each tuples has one keyword. Split the attribute Holdings into two attributes, DateFrom and DateTo.
 
2NF: Split R into
R1(Title, DateFrom, DateTo)
R2(Title, Keyword)
 
Repeating the title of a journal in many tuples is potentially a waste of space and might be trying to update. There is a question of policy here: if a journal changes its title, does one keep the backissues with their original title? The answer is generally yes. So we do not need to worry about updates, though we should note that the current design fails to retrieve all holdings of a journal which has changed its name.
 
We could avoid the repeated title and save space by using:
R1(TitleID, DateFrom, DateTo)
R2(TitleID, Keyword)
R3(TitleID, Title)
It is possible, however, that the cost in time spent in doing the join for almost every query might outweigh the savings in space.
 
We could also eliminate the confusion of keywords by introducing a Keyword relation:
R1(TitleID, DateFrom, DateTo)
R2(TitleID, KeywordID)
R3(TitleID, Title)
R4(KeywordID, Keyword)

b)      Are there any other alternate keys?
ANS:
As the example shows, neither B, nor C, nor D is an alternate key. 

AB, AC, and AD are not minimal, so they are not alternate keys. Since BC is an alternate key and (D→C), we see that BD is also an alternate key.
From the example, we find that CD is not an alternate key.
 
Each set of three attributes, BCD, ACD, ABD, and ABC, properly contain an alternate key and so are not themselves alternate keys. Similarly for ABCD.
 
So, BD is the only other alternate key.

ANS:
 Yes. If we insert a row (5, c, 9, x), then A and BC are still candidate keys, but (D→C) no longer holds.

ANS:
Yes. If we update row (1, a, 7, x) to (1, a, 9, x), again DC will be violated.

e)        Is there a deletion anomaly?

Yes. If we delete row (3, a, 8, y), then the association (8, y) of C and D will be lost.

g)      Is there an insertion anomaly?
ANS:
Yes.The delicate point is the functional dependency FD(BC, D). Perhaps we can have two tuples with the same BC values but different D values? Looking at the first row of R, (1, a, 7, x), we aim at a tuple (5, a, 7, q). This tuple comes from inserting (5, a, q) into R1 and (7, q) into R2.

h)        Is there an update anomaly?
ANS:
Yes. It is possible to update (8, y) in R2 to (7, y). Then when we create R from the join of R1 and R2, BC will no longer be a candidate key.

i)        Is there a deletion anomaly?
ANS:
 No. Deleting a row from R1 or R2 will not lead to unexpected data loss. There is the issue of whether to cascade the deletions from R2 to R1, but this does not count as an anomaly.

6.      Suppose we have a relational schema R(A,B,C,D,E) for which the following functional dependencies hold:
A→C,E
B→D,
BC→E,
DE→B
 
a)      Find all the candidate keys for R.
ANS:
Since A is not determined by any of the given FD's, any candidate key must contain A. Taking each pair as a determinant:
A → ACE
AB → ABCE→ ABCDE
AC → ACE
AD → ACDE → ABCDE
AE → ACE
So AB and AD are the only candidate keys.
 
There are no others. A candidate key cannot properly contain AB or AD. The only attribute set that fits these conditions is ACE, which does not determine B or E.

b)      Show that R is not in 2NF.
ANS:
If we choose either AB or AD as primary key, then (A → C,E) violates 2NF.

c)      Propose a lossless join decomposition of R that puts it into 3NF.
ANS:
One possibility is to first factor out B→D, second factor out A→C,E:
R(ABCDE) = R1(BD) joinB R2(ABCE)
= R1(BD) joinB ( R3(AB) joinA R4(ACE) )

d)      Does your splitting preserve dependencies? (Do not worry if it does not!)
ANS:
No. (A→C,E) and (B→D) are preserved, but (BC→E) and (DE→B) are broken.
