Advantages of SQL

  • Satisfies ideals for database language
  • (Relatively) Easy to learn
  • Portability
  • SQL standard exists
  • Both interactive and embedded access
  • Can be used by specialist and non-specialist.

Disadvantages of SQL

  • Impedance mismatch - mixing programming paradigms with embedded access
  • Lack of orthogonality - many different ways to express some queries
  • Language is becoming enormous (SQL-92 is 6 times larger than predecessor)
  • Handling of nulls in aggregate functions
  • Result tables are not strictly relational - can contain duplicate tuples, imposes an ordering on both columns and rows.

These commands always have the potential to put the database into an inconsistent state. For example, if a row of a relation is deleted, what happens to rows in other relations that have foreign keys which referenced that row? If the foreign key is a primary key, then it will be necessary to trigger a cascading delete, otherwise it will be sufficient to nullify the foreign key.

The SELECT Statement

The general structure of an SQL SELECT statement is: 

  SELECT [DISTINCT | ALL] <column names, functions, aggregate functions>
        FROM <table names>
        WHERE <conditions>
        GROUP BY <column names>
        HAVING <conditions on aggregated columns>
        ORDER BY <subset of column names> [ASC | DESC]

 A SELECT statement returns a result set, not a relation. In a result set, the rows and columns are ordered. Furthermore, a result set may contain duplicate rows. To obtain a result set without duplicate rows, an explicit DISTINCT qualifier must follow the SELECT. When modeling queries with relational calculus, the projection operator must be modified to permit duplicate tuples and a sort operator must be added. 

A result set can be treated as a relation for most purposes. In particular, SELECT statements can be nested. More generally, a view is defined by a query. It is the result set obtained by running the query on a database instance. A view can be used in SQL where a relation is expected.

Processing a Query

Conceptually, the sequence of operations that defines the result set is roughly as follows:

  •         Form a Cartesian product of all the tables in the FROM clause
  •         Eliminate all rows that do not satisfy the WHERE clause
  •         Add columns for the simple functions
  •         Compute the simple functions
  •         Sort the remaining rows on the column names in the GROUP BY clause
  •         Add columns for any aggregate functions in the SELECT and HAVING clauses
  •         Compute the aggregate functions
  •         Eliminate the rows that do not satisfy the HAVING clause
  •         Project onto the requested columns and aggregate functions
  •         Sort as specified by the ORDER BY clause

 This sequence of operations is far from optimal. It just helps us to visualize what the functional meaning of each clause of a query.
 In a decent DBMS, query optimization is automatic, and rewriting SQL will not improve performance.

Aggregate Functions

MAX, MIN, SUM, AVG apply to columns.

COUNT applies to result sets. 

In

SELECT * FROM ...

the * stands for the whole result set. Thus, in

SELECT COUNT(*) FROM ...

the aggregate function COUNT counts all the rows in the result set indicated by *.

Expression Operators

attribute IN result set

attribute < SOME column (same as ANY)

attribute < ALL column

EXISTS result set

result set UNION result set

result set INTERSECT result set (not supported by Access)

result set EXCEPT result set (not supported by Access)

Subqueries

        A scalar query returns a result set with a single row and column

        A row query returns a result set with a single row

        A column query returns a result set with a single column

        A table query returns a result set with multiple rows and at least one column

 Each type of query can be used as a subquery (nested select) in a WHERE or HAVING clause whenever the dimensions of the result set match the requirements of SQL syntax.

 Examples of SELECT Statements

1.      What is in the Part table?

Select *
FROM Part

 2.      How much does a Cog cost?

SELECT Cost
FROM Part
WHERE Name = 'Cog'

 3.      Does any part not have a name?

SELECT PID
FROM Part
WHERE Name IS NULL

 4.      List by PID and WID the quantity of each part for which the quantity is at least 25. Order the result with quantity in descending order.

Using ORDER BY to sort results

SELECT PID, WID, Quantity
FROM WP
WHERE Quantity >= 25
ORDER BY Quantity DESC


Joins

 5.      List by WID the quantities of part PID=1 in LA warehouses.
       Using Cartesian product to compute a join

        SELECT WID, Quantity
        FROM Warehouse, WP
        WHERE
            Warehouse.WID = WP.WID AND
            WP.PID = 1 AND
            Warehouse.Address LIKE 'LA' (LIKE '%LA%', or '*LA*' in Access)

 6.      List by WID the quantities of part PID=1 in LA warehouses.
       Using column subqueries and IN to do a join
       
        SELECT WID, Quantity
        FROM WP
        WHERE
            PID = 1 AND
            WID IN (
                Select WID FROM Warehouse WHERE Address LIKE '%LA%'
                )

7.      What parts are in no warehouse?

        SELECT PID, Name
        FROM Part
        WHERE
            PID NOT IN (
                SELECT PID
                FROM WP
                WHERE Quantity > 0
            ) 

8.      Which parts do we have in LA warehouses?

Using a table subquery and EXISTS to do a join

SELECT DISTINCT PID
FROM WP
WHERE EXISTS (
                SELECT *
                FROM Warehouse AS W
                WHERE
                    W.WID = WID AND
                    Address LIKE '%LA%'
 )

Try to modify the above query to display the part name as well.

9.  [Optional] Create a view that doesn't show quantity for PID=1

Using UNION on result sets

 CREATE VIEW  WPView  AS
    SELECT WID, PID, NULL AS Quantity 
    FROM WP
    WHERE PID = 1
        UNION
    SELECT WID, PID, Quantity
    FROM WP
    WHERE PID <> 1

Aggregate functions

 11.  What is the maximum quantity of any part in any warehouse?

SELECT MAX(Quantity)
FROM WP

 12.  Which are the PID(s) with maximum quantity in some warehouse?

Using a scalar subquery 

SELECT PID
FROM WP
WHERE
    Quantity = ( SELECT MAX(Quantity) FROM WP )

 13.  What is the total quantity for each part across all warehouses?

Using GROUP BY to guide aggregation

SELECT PID, SUM(Quantity) AS [Total Quantity]
FROM WP
GROUP BY PID

 14.  What is the total quantity for each part across all warehouses with total quantity > 100?

Using HAVING to constraint aggregates

SELECT PID, sum(Quantity) AS [Total Quantity]
FROM WP
GROUP BY PID
HAVING sum(Quantity) > 100

 15.  What part do we have in greatest quantity?

Using ALL as a replacement for MAX in an expression

SELECT PID, sum(Quantity) AS [Max Quantity]
FROM WP
GROUP BY PID
HAVING sum(Quantity) >= ALL (
            SELECT sum(Quantity)
            FROM WP
            GROUP BY PID )

 16.  List by warehouse address the total number of parts in the warehouse and the total number of parts which have quantity at most 50 in that warehouse.

 Part 1: Count of parts by warehouse address
 
       SELECT Address, sum(Quantity) AS [Count of Parts]
       FROM Warehouse AS W, WP
       WHERE
            WP.WID = W.WID
       GROUP BY Address  

Part 2: Count of parts with quantity at most 50 by warehouse address

SELECT Address, sum(Quantity) AS [Count of Limited Parts]
FROM Warehouse AS W, WP
WHERE
    WP.WID = W.WID AND
    Quantity <= 50
GROUP BY Address

 Part 3: Combining Part 1 and Part 2 by address

Wrong answer (doing join first, then select): 

SELECT Address, sum(WP.Quantity) AS [Count of Parts],
    sum(WP2.Quantity) AS [Count of Limited Parts]
FROM Warehouse AS W, WP, WP AS WP2
WHERE
    WP.WID = W.WID AND
    WP2.WID = W.WID AND
    WP2.Quantity <= 50
GROUP BY Address

 Incomplete answer (doing selects first, then join):

SELECT Q1.Address, pno AS [Parts], lpno AS [Limited Parts]
FROM (
    SELECT Address, sum(Quantity) AS [pno]
    FROM Warehouse AS W, WP
    WHERE
        WP.WID = W.WID
    GROUP BY Address
    ) AS Q1,
    (
    SELECT Address, sum(Quantity) AS [lpno]
    FROM Warehouse AS W, WP
    WHERE
        WP.WID = W.WID AND
        Quantity <= 50
     GROUP BY Address
    ) AS Q2
WHERE
    Q2.Address = Q1.Address

 Complete answer (including addresses with no quantities less than 51):

SELECT Q1.Address, pno AS [Parts], lpno AS [Limited Parts]
FROM (
    SELECT Address, sum(Quantity) AS [pno]
    FROM Warehouse AS W, WP
    WHERE
        WP.WID = W.WID
    GROUP BY Address
    ) AS Q1,
    (
    SELECT Address, sum(Quantity) AS [lpno]
    FROM Warehouse AS W, WP
    WHERE
        WP.WID = W.WID AND
        Quantity <= 50
    GROUP BY Address
    UNION
    SELECT Address, NULL AS [lpno]
    FROM Warehouse AS W, WP
    WHERE
        WP.WID = W.WID AND
        NOT EXISTS (
                SELECT *
                FROM WP
                WHERE
                   WP.WID = W.WID AND
                    Quantity <= 50
                )
        ) AS Q2
  WHERE
        Q2.Address = Q1.Address

UNION here could be replaced by an OUTER JOIN:

SELECT Q1.Address, pno AS [Parts], lpno AS [Limited Parts]
FROM (
    SELECT Address, sum(Quantity) AS [pno]
    FROM Warehouse AS W, WP
    WHERE
        WP.WID = W.WID
    GROUP BY Address
    ) AS Q1,
    (
    SELECT Address, sum(Quantity) AS [lpno]
    FROM Warehouse LEFT JOIN WP ON (
        WP.WID = Warehouse.WID AND
        Quantity <= 50
    )
GROUP BY Address
) AS Q2
WHERE
    Q2.Address = Q1.Address
