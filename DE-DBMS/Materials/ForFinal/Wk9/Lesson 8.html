CS422:  Database Management Systems
 
Lesson 8 - Logical Database Design

Purpose : -  Ability to derive a logical model from a conceptual model by (1) refining an ER diagram, and (2) deriving a relational model from the refined ER diagram.

Design Methodology

A structured approach that uses procedures, techniques, tools, and documentation aids to support and facilitate the process of design.

 

Three stages of design

  1.  Conceptual Database Design
The process of constructing a model of the information used in an enterprise, independent of all physical considerations. At this stage you haven’t yet decided on data             model yet. You are on conceptual level only.  You’ve only created an ER model at this stage.

  1. Logical Database Design

The process of constructing a model of the information used in an enterprise based on a specific data model (e.g. relational), but independent of a particular DBMS and other physical considerations.

Data model is decided at this stage.  Data model means how data is represented in your mind.  E.g when we say that relational data model will be used, we are saying that data will be kept in the form of tables.  So at this stage, relations are decided and created.

  1. Physical Database Design
The process of producing a description of the implementation of the database on secondary storage; it describes the storage structures and access methods used to achieve efficient access to the data.

Basically at this stage we decide on s/w and h/w.  Also on which platform it’ll be run, how much disk space will be needed etc.  All kinds of such decisions are taken at this level.

 

First Stage - Conceptual Database Design

Build local conceptual data model for each user view
  1. Identify entity types
  2. Identify relationship types
  3. Identify and associate attributes with entity or relationship types
  4. Determine attribute domains
  5. Determine candidate and primary key attributes
  6. Specialize/generalize entity types (optional step)
  7. Check model for redundancy
  8. Validate conceptual model against user transactions
  9. Review Local Conceptual Data Model with User

 

Second Stage - Logical Database Design for the Relational Model

  1. Map Local Conceptual Data Model to Local Logical Data Model
  2. Derive Relations from Local Logical Data Model
  3. Validate Model using Normalization
  4. Validate Model against User Transactions
  5. Draw Entity-Relationship Diagram
  6. Define Integrity Constraints
  7. Review Local Logical Data Model with User 

At some stage, the local models must be integrated into a global model. One can either:

        integrate the local conceptual models into a global conceptual model and then derive a global logical model from that, or
        derive local logical models and then integrate these into a global logical model.

In our case, as we are studying relational database, our logical model will be tables.  There are various rules for creating tables from ERD.

Mapping from the ER model to the relational model

  •  Handle Entities

        An entity usually becomes a table.

        The primary key of a table corresponding to a strong entity is the same as the primary key for the entity.

        When to use system generated key?

        A weak entity will get some or all of its key attributes from the corresponding strong entity (or entities).

        All entities in a hierarchy have the same primary key.
 

  • Handle relationships

        For 1-N relationship, add a foreign key on the N side referring to a primary key on the 1 side.

    For 1-1 relationsihp :

Mandatory participation on both sides of 1:1 relationship - Combine entities involved into one relation and choose one of the primary keys of original entities to be primary key of the new relation, while the other (if one exists) is used as an alternate key.

Mandatory participation on one side of a 1:1 relationship - Entity with optional participation in relationship is designated as parent entity, and entity with mandatory                  participation is designated as child entity. A copy of primary key of the parent entity is placed in the relation representing the child entity. If the relationship has one or             more attributes, these attributes should follow the posting of the primary key to the child relation.
Note that there's a mistake by Prof. Nair in the video (CS422-06-04) while identifying parent and child entities in the example of Employee and Project. 

Optional participation on both sides of a 1:1 relationship - the designation of the parent and child entities is arbitrary unless we can find out more about the relationship that can help a decision to be made one way or the other. So there's no rule for this.

   For 1-1 recursive relationship, follow the rules for 1-1 relationship.

   For Many-Many relationship: It is not possible to implement M:M in relational database. There is a need to add a 3rd table.  Then the PK of this new table (entity) will be made up of the PK's of the other 2 tables which are involved in the M:M relationship. 
  • Multivalued attributes - Create a new relation to represent multi-valued attribute and include primary key of entity in new relation, to act as a foreignkey.
  • Follow the steps of normalization for the relational model

        Determine all functional dependencies.

        Test for conformity to 2NF, 3NF, and BCNF, making lossless join decompositions as necessary.

        4NF verify that insertion of a row in a table does require assigning arbitrary values to some columns to satisfy entity integrity.

 If you find that your tables are not even in 2NF then you missed some of the M:M relationships in the ERD. If your tables are in 2NF but not in 3NF then you missed some of the 1:M relationships.

  • Validate against transactions

        Create sample database and write all transactions to verify that all necessary "relationships" are represented in the database.
