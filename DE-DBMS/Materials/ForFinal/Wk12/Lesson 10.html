Purpose : - To understand recovery management and the function of log files, checkpoints, and recovery protocols; to be able to use precedence diagrams, wait-for graphs, and dirty-read graphs.


Recoverability

How to return the database to a consistent state after a failure, such as transaction failure, power failure, or the DBMS crashes.

 

Recoverable Schedule

A schedule is recoverable when, for each pair of transactions T1 and T2, if T2 reads a data item after T1 has written it, then T1 commits before T2.

 
A recoverable schedule is free from uncommitted dependency problems. A recoverable schedule makes it possible to recover from transaction failure by cascading rollback.

 

Not recoverable

T1: R(x) W(2*x) ...

T2: R(x) R(y) Commit

 

Recoverable

T1: R(x) W(2*x) Commit

T2: R(x) R(y) ...

 

 Dirty-Read Graph

If we construct a graph in which each transaction is represented by a vertex and each dirty read is represented by a directed edge from the transaction which wrote to the transaction which read, then a cycle in the dirty-read graph indicates that no sequence of commits will yield a recoverable schedule.

 

Cascadeless Schedule

Never read dirty dataT2.R(x) cannot come between T1.W(x) and T1.COMMIT

 

T1: R(x) W(2*x) Commit

T2: R(y) R(x) Commit

 

Cascadeless schedules aim to improve performance by eliminating cascading rollbacks. They are not necessary for database integrity.

 

Recovery Control

Need for recovery control

  • system crashes wipe out volatile storage (main memory)
  • transaction failure or rollback

 

Transactions

  • Atomicity requires effects of a failed transaction to be undone.
  • Durability requires effects of a committed transaction to endure.

 

Recovery

  •         Durability: If failure occurs between commit and database buffers being flushed to secondary storage, then recovery manager has to redo (rollforward) transaction's updates.
  •         Atomicity: If transaction had not committed at failure time, recovery manager has to undo (rollback) any effects of that transaction.

 

Transactions represent basic unit of recovery.

 

Recovery Manager

If database has been damaged:

  • restore last backup copy of database
  • reapply updates of committed transactions using log file

 

If database is only inconsistent:

  • undo changes that caused inconsistency (how to find these?)
  • redo transactions as necessary to ensure updates reach secondary storage

 

Components of recovery management

  • transaction log file
  • checkpoints
  • recovery protocol

 

Log File

Contains information about all updates to database:

  • transaction records
  • checkpoint records

 
Often used for other purposes (for example, auditing). A business might generate 10 GB of log file daily.

 

Checkpoint

  • Point of synchronization between database and log file:
  • write log records in main memory to secondary storage
  • write modified blocks in database buffers to secondary storage
  • write a checkpoint record to the log file identifying currently active transactions
  • When failure occurs, redo all transactions that committed since the checkpoint and undo all transactions active at time of crash.

 
Make perhaps three or four checkpoints per hour.

 

Protocols

Deferred Update Protocol

  • When a transaction starts, write a transaction start record to the log
  • Write all transaction operations to log
  • When about to commit, write commit record to log, write log to disk, then commit
  • Use log records to perform actual update of database for committed transactions

After a failure

  • Access most recent checkpoint record
  • Redo committed transactions listed as active
  • Redo committed transactions that started after the checkpoint

 

Immediate Update Protocol

  • When transaction starts, write a transaction start record to the log
  • Record each write operation in the log, then perform it on the database buffers
  • When about to commit, write commit record to log, write log to disk, then commit

After a failure

  • Access most recent checkpoint record
  • Use the after-image of the write records to redo
    • any committed active transaction
    • any committed transaction that started after the checkpoint
  • Use the before-image of the write records to undo any uncommitted transaction
